> AUROC vs F1, Threshold 설계, 그리고 재현성 검증

* * *

## 들어가며

FDS 모델을 개발하면서 감사팀으로부터 가장 많이 받은 질문은 이것이었다.

> "왜 AUROC가 아니라 F1으로 모델을 선택했는가?" "Threshold는 어떻게 정했는가?" "결과가 데이터 분할에 따라 달라지는 것 아닌가?"

이 질문들은 단순한 형식적 검증이 아니다. 극단적인 class imbalance(양성 0.5%) 환경에서는 **' 좋아 보이는 수치'와 실제 운영 성능이 다를 수 있기 때문**이다.

이 글에서는 다음 세 가지를 정리한다.

  1. **AUROC vs F1** — 무엇을 기준으로 모델을 선택해야 하는가
  2. **Threshold** — 어떻게 결정해야 하는가
  3. **재현성 검증** — 데이터 분할에 따른 성능 변동을 어떻게 다루는가



* * *

## 1\. AUROC가 높으면 좋은 모델인가?

### 1.1 AUROC의 장점

AUROC는 **threshold-free metric** 이다. 어떤 임계값을 설정하느냐와 무관하게, 모델이 양성과 음성을 전반적으로 얼마나 잘 분리하는지를 평가한다.

$$AUROC = P(\hat{y}_{pos} > \hat{y}_{neg})$$

구체적으로는, 임의로 추출한 양성-음성 쌍에서 모델이 양성에게 더 높은 점수를 부여할 확률이다.

이 특성 덕분에 AUROC는 다음 용도에 적합하다.

  * 임계값 설정 전 단계에서 모델의 전반적인 분리력(ranking quality) 확인
  * 서로 다른 모델 간 순위 보존 능력 비교
  * 학술적으로 표준화된 baseline 지표



### 1.2 그러나 양성 비율 0.5% 데이터에서는 문제가 생긴다

ROC 곡선의 x축은 FPR(False Positive Rate)이다.

$$FPR = \frac{FP}{FP + TN}$$

양성이 0.5%라면 음성은 99.5%다. 분모인 $FP + TN$, 즉 실제 음성의 수가 압도적으로 많기 때문에, **FP가 꽤 발생해도 FPR은 낮게 유지** 된다. 결과적으로 ROC 곡선이 낙관적으로 그려지고, AUROC가 높게 측정된다.

하지만 실제 운영에서 문제가 되는 것은 이렇다.

  * AUROC는 높지만 **Precision은 낮을 수 있다**
  * Precision이 낮다는 것은, 탐지로 올라온 건수 중 실제 이상이 적다는 의미다
  * FDS 환경에서 **False Positive = 조사 리소스 낭비** , **False Negative = 미탐지**



이 문제는 AUPRC(Area Under the Precision-Recall Curve)를 보완 지표로 함께 검토하면 더 명확해진다. PR 곡선은 x축이 Recall, y축이 Precision으로 구성되어 음성 클래스 수에 영향을 받지 않기 때문에, 불균형 데이터에서 AUROC보다 더 민감하게 모델 간 차이를 드러낸다.

### 1.3 왜 F1을 기준으로 선택했는가

F1은 Precision과 Recall의 조화평균이다.

$$F_1 = \frac{2 \cdot Precision \cdot Recall}{Precision + Recall}$$

산술평균이 아닌 조화평균을 쓰는 이유는, **두 값 중 하나가 낮으면 전체 점수가 크게 낮아지기 때문** 이다. 즉 F1은 다음을 강제한다.

  * Precision이 낮으면 점수 하락
  * Recall이 낮으면 점수 하락
  * **한쪽만 좋은 모델은 선택되지 않는다**



0.5% 환경에서는 이 균형 강제가 특히 중요하다.

  * Recall만 높이면 → FP 폭증 → 조사 리소스 낭비
  * Precision만 높이면 → FN 폭증 → 실제 이상 미탐지



F1은 이 trade-off를 하나의 숫자로 압축하는 지표다.

### 1.4 결론: AUROC는 "참고", F1은 "의사결정"

실무에서는 다음 순서로 접근했다.

  1. **AUROC** — 모델의 전반적 분리력 확인 (sanity check)
  2. **AUPRC** — 불균형 환경에서 보완적 분리력 검토
  3. **F1** — 최종 모델 선택 기준 (운영 관점 decision metric)

## 2\. Threshold는 어떻게 결정했는가?

### 2.1 흔한 오해: "F1을 최대화했다"

많은 경우 threshold를 단순히 **F1이 가장 높은 지점** 에서 선택한다. 하지만 실제 운영 환경에서 이 방식이 항상 적절하지 않다.

이유는 다음과 같다.

  * 조사 가능한 리소스(월별 처리 가능 건수)는 제한되어 있다
  * False Positive의 비용은 단순한 수치가 아니라 실제 인력과 시간이다
  * Metric 최대값이 운영 제약과 일치하지 않을 수 있다



### 2.2 실제 접근 방식

Threshold 결정은 다음 네 가지 요소를 동시에 고려했다.

  1. **Validation set 기반 평가** — test set에 대한 leakage 없이 threshold 탐색
  2. **Precision–Recall curve 분석** — 전체 threshold 범위에서의 trade-off 시각화
  3. **비즈니스 수용 가능한 Precision 수준 확인** — 운영팀이 허용 가능한 오탐률 기준 설정
  4. **처리 가능 물량(capacity) 범위 내 유지** — 탐지 건수가 운영 처리량을 초과하지 않도록 제약



즉, **" Metric을 최대화하는 값"이 아니라 "비즈니스적으로 수용 가능한 균형점"** 을 찾는 과정이었다. 이는 cost-sensitive decision boundary 설계와 맥락을 같이 한다.

### 2.3 왜 이것이 중요한가

모델은 점수를 내는 시스템이고, 비즈니스는 의사결정을 내리는 시스템이다. **Threshold는 이 둘을 연결하는 레이어** 다.

따라서 threshold 설정에는 다음 네 가지가 동시에 반영되어야 한다.

  * 모델 성능 (F1, Precision, Recall)
  * 리소스 제약 (처리 가능 건수)
  * 오탐 비용 (False Positive cost)
  * 미탐 비용 (False Negative cost)

## 3\. 데이터 분할과 재현성 검증

모델이 진짜 좋은지 판단하려면 **" 운 좋게 나온 결과"인지 검증해야 한다**.

### 3.1 데이터 구성

구분 | 용도  
---|---  
Train (80%) | 모델 학습  
Dev (10%) | Threshold 탐색 및 모델 선택  
Test (10%) | 최종 성능 평가 (1회만 사용)  
  
Dev set을 threshold 탐색 전용으로 분리한 이유는, **threshold 최적화 과정에서 test set에 대한 간접적 leakage가 발생하는 것을 방지** 하기 위해서다.

### 3.2 Seed 민감도 검증

단일 split 결과는 우연일 수 있다. 특히 양성이 0.5%라면, Dev/Test에 포함된 양성 샘플 수 자체가 수십~수백 건에 불과하다. 이 소수의 샘플이 어떻게 배치되느냐에 따라 F1이 크게 흔들릴 수 있다.

따라서 random seed를 바꾸어 총 3회 실험을 수행했다.
    
    
    Data_1: seed = 42  (기본 split)
    Data_2: seed = 123
    Data_3: seed = 999

각 split에서 확인한 항목은 다음과 같다.

  * AUROC 안정성 (split 간 표준편차)
  * F1 변동 폭
  * 최적 threshold의 일관성



> **왜 k-fold를 사용하지 않았는가?** 본 프로젝트 데이터는 시계열 구조를 가지므로, random k-fold를 적용하면 미래 데이터가 훈련에 포함되는 **시간 누수(temporal leakage)** 가 발생한다. 이를 방지하기 위해 시간 순서를 유지한 hold-out 방식으로 3회 반복 검증을 수행했다.

### 3.3 왜 이 검증이 중요한가

Imbalanced data에서는 소수 클래스 샘플의 배치 하나가 결과를 크게 바꿀 수 있다. 3회 반복에서 성능이 일관되게 유지된다면, 모델 선택이 특정 split의 우연에 의존하지 않았다는 근거가 된다. 반대로 변동이 크다면, 모델이나 피처 설계를 재검토해야 한다는 신호다.

* * *

## 정리

FDS와 같은 극단적 불균형 환경에서 모델 선택은 단순한 metric 비교가 아니다.

결국 중요한 질문은 이것이다.

> **" 이 모델이 실제 운영에서 의미 있게 작동하는가?"**

좋은 모델은 높은 AUROC를 가진 모델이 아니라, **운영 제약과 비용 구조 안에서 지속 가능하게 작동하는 모델** 이다.
